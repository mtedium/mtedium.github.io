【故障复盘】WSL2 镜像模式下 apt update 报 No route to host (113) 的精准排查与修复

在将 WSL2 的网络模式升级到最新的 Mirrored（镜像模式） 后，不少开发者会遇到一个诡异的现象：在 Ubuntu 中执行 sudo apt update 或 curl 外网时，直接卡死并抛出如下错误：

Plaintext

很多网上的教程会让你盲目地去修改 /etc/resolv.conf、换国内源或者疯狂重启。本文将带你拒绝瞎猜，通过底层数据流精准定位这一类 WSL 镜像模式网络死锁的根本原因。

一、 现象分析：DNS 没挂，是路断了

首先，观察 apt update 的报错核心：

这里有一个非常重要的细节：WSL 已经把 mirrors.aliyun.com 成功解析成了 IPv4 地址（222.192.187.244）。

这铁证如山地表明：系统的 DNS 解析是完全正常的，问题绝不在 /etc/resolv.conf 上。

真正的罪魁祸首是 113: No route to host（目标主机不可达）。这意味着 Linux 内核在拿到目标 IP 后，打算把数据包丢给网关发出去，但本地的网络协议栈或路由表直接回绝了它：“我找不到通往这个 IP 的路。”

二、 底层排查：抓出那个“死网关”

为了看清数据包到底卡在哪里，我们在 WSL 终端中运行了以下两条命令：

1. 查看 Linux 路由表

可以看到，当前 WSL 的默认路由（default）将所有外网流量都指向了一个网关：10.210.71.254。

2. 测试网关连通性

既然所有流量都要走 10.210.71.254，那这个网关真的活着吗？我们直接 ping 它：

Bash

致命原因找到了：Destination Host Unreachable。

Windows 给 WSL 分配了一个根本不存在的“死网关”。Linux 尝试把所有上网流量发给它，结果连门都出不去，直接引发了 No route to host。

三、 深度根源：为什么会出现死网关？

为什么在 Mirrored（镜像模式）下会出现这种死网关？

原因在于 WSL 的高级设置中开启了 已启用自动代理 (autoProxy = true)。

在镜像模式下，WSL 与 Windows 共享同一个网络栈。当开启自动代理时，Windows 会在后台强行在本地生成一系列动态路由，试图将 WSL 的流量拦截并转发给宿主机的代理软件。

然而，如果此时：

  1. Windows 端的代理软件（如 Clash、v2rayN、Sing-box）没有开启 TUN 模式（虚拟网卡模式）；
  2. 或者是代理软件断开、电脑休眠唤醒后，Windows 的网络套接字（Winsock）发生死锁；

就会导致 Windows 端的“接盘进程”挂了，但分配给 WSL 的动态路由和代理网关却没有被清理。WSL 只能傻傻地把流量往死网关里送。

四、 彻底解决方案

既然知道了是 “Windows 代理策略/网络栈污染” + “WSL 镜像模式” 导致的网卡死锁，我们就需要从 Windows 宿主机层面进行彻底清洗。

第一步：关闭 WSL 自动代理

打开你的 WSL 管理工具（或直接编辑 C:\Users\<你的用户名>\.wslconfig），找到网络设置：

  • 将 已启用自动代理 (autoProxy) 拨至 关闭 状态。
  • （可选）如果必须要开代理，建议在 Windows 代理软件中开启 TUN 模式，而不是依赖 WSL 的 autoProxy。

第二步：管理员身份清洗 Windows 网络栈

由于错误的套接字和路由可能残留在 Windows 内存中，我们需要强刷网络：

  1. 在 Windows 搜索框输入 cmd，右键选择 以管理员身份运行。
  2. 依次输入并执行以下命令：

PowerShell

第三步：重启电脑（必须）

netsh winsock reset 涉及到 Windows 核心网络驱动层的重置，必须重启电脑才能释放内存中的死路由。

第四步：验证

Bash

此时，镜像模式会干净地抓取你 Windows 当前真实的物理网卡网关，网络瞬间秒通！

五、 总结与建议

  • 拒绝盲目试错：遇到网络问题，先看 apt 能不能解析出 IP（判断 DNS），再看 ip route 和 ping 网关（判断路由）。
  • 镜像模式避坑：WSL2 的 Mirrored 模式非常好用，但极度依赖 Windows 宿主机网络环境的纯净度。如果频繁使用代理，强烈建议开启代理软件的 TUN 模式，并关闭 WSL 自带的 autoProxy 选项，以防网卡死锁。
